# FairGuard — Agentic Investigation Submission

**Team:** FairGuard — Mokshit Surana, Archit Rathod
**Challenge:** Northwestern GAIN Agentic AI Investigative Journalism Challenge
**Reproducibility:** self-contained (see *Reproducing* below). Source also mirrored at https://github.com/MokshitSurana/northwestern-challenge (not required for this submission).
**Live demo (optional):** the Reporter UI rendered on the shipped findings — https://fair-guard-ai.vercel.app/ — a static snapshot of the last pipeline run, offered so a reviewer can browse the four-gate dashboards, money trails, and provenance without cloning. Not part of the submission package and not required to evaluate or reproduce it; the authoritative artifacts are the skills + records in this zip.
**License:** MIT

FairGuard is a suite of nine reusable Claude Code **Agent Skills** for investigative
journalism, plus the newsworthy findings they produced on the provided corpus of federal
Lobbying Disclosure Act (LDA) filings and congressional press releases (2022–Q1 2026). The
skills index the corpus, resolve entities, detect former officials now lobbying the agency
they once served, follow the federal money to those officials' clients, corroborate against
congressional press releases, compose a conflict-of-interest graph, and manage the
reportability workflow (requests for comment, citation archiving).

---

## 1. Included skills

Each skill is self-contained under `skills/<name>/`: a `SKILL.md` (portable, standard Agent
Skill format) plus the `scripts/` it runs, and `references/`, `assets/`, or `cases/` where the
skill uses them. Scripts assume the package root as the working directory (they read/write
`output/`, `web/public/`, `notes/`), so invoke them from the package root as
`uv run skills/<name>/scripts/<script>.py`.

| Skill (`SKILL.md`) | What it does | Supports finding(s) |
|--------------------|--------------|---------------------|
| `skills/doctor/` | Cross-platform setup validator + guided onboarding | Reproducibility gate (all findings) |
| `skills/lda-corpus-indexer/` | ETL: raw LDA/press-release dumps → Parquet → DuckDB (handles 7 documented silent-data-loss parser quirks) | Data substrate for **Findings 01 & 02** |
| `skills/entity-resolver/` | Normalizes org/person names (held-out F1 = 0.963) | Candidate ranking in **Finding 02** |
| `skills/revolving-door-detector/` | Ranks former senior officials by concentration of lobbying back on their former agency | **Finding 01** (Bridenstine/Artemis anchor) and **Finding 02** (structural pattern) |
| `skills/federal-award-tracer/` | Traces USAspending.gov awards from an agency to a lobbyist's clients; splits discretionary grants from routine program dollars | **Finding 02 money trail** |
| `skills/press-release-cross-ref/` | Cross-references candidates against congressional press releases | Corroboration in **Finding 02** |
| `skills/coi-graph/` | Composes scan + trace + pressrel into a conflict-of-interest graph | **Finding 02** synthesis |
| `skills/comment-tracker/` | Request-for-comment workflow state machine | Reportability gate 4 (`notes/09`) |
| `skills/archive-on-cite/` | Snapshots every cited URL to Wayback + Archive.today | Citation durability (all findings) |

## 2. Findings

The written report is **`findings-report.md`** (also as `findings-report.pdf`). Supporting,
record-sourced verification notes are in `notes/`:

- **Finding 01 — The Artemis Group (anchor, verified).** A lobbying firm founded by former
  NASA Administrator Jim Bridenstine directs ~39% of its filings at NASA, staffed by former
  Bridenstine officials. Numbers verified against corrected House data (133 Senate filings,
  52 targeting NASA; 115 House filings; 125 House lobbyist rows naming Bridenstine). Sourced
  in `notes/05_finding_bridenstine.md`.
- **Finding 02 — Structural revolving-door pattern (Track 2).** A ranked list of 139
  comparable candidates across 22 federal agencies (`notes/06_structural_pattern_findings.md`,
  auto-generated by the scan). The top candidates are independently role-verified and carry
  large federal money trails: **Steinberg → DOE ≈ $1.08B** in discretionary battery-supply-chain
  grants to clients (reproduces to the dollar: $1,080,820,046); **Limbaugh → Interior $133.1M**
  in Bureau of Reclamation grants (the shipped grants-only case; Glenn-Colusa's ~$25.6M
  Reclamation award is reported separately — see `notes/08`); and the
  USDA cases (mostly routine program dollars, with Global Clean Energy's $32.3M NRCS grant the
  discretionary standout). Verification and money trail: `notes/08_external_verification_top_candidates.md`.
  Legality + fair-comment analysis: `notes/09_reportability_gates_207_and_comment.md`.

Each finding is framed as conflict-of-interest **structure and stakes**, not proven wrongdoing.

## 3. Traces

All interaction traces are in `traces/`, indexed by **`traces/README.md`**, which maps every
trace to the skill invocation and finding it supports. It contains:

- **`trace-01`–`trace-02`** — narrative sessions (setup; parser-bug debugging).
- **`trace-03`–`trace-11`** — one curated, rendered transcript per skill.
- **Three raw JSONL logs:** two are **live reproduction runs** —
  `trace-raw-repro-scan-finding02.jsonl` (regenerates Findings 01 & 02) and
  `trace-raw-repro-trace-moneytrail.jsonl` (regenerates the money trail against the live
  USAspending API, Steinberg → DOE $1,080,820,046 exact) — plus
  `trace-raw-session-tracev1.1-evals-and-gates.jsonl`, a full working session (the
  `federal-award-tracer` eval loop and the §207 / request-for-comment gate closure).

`traces/README.md` discloses honestly why the original build sessions aren't retained as raw
logs (Claude Code keeps them only locally, briefly), notes that the reproduction logs are the
strongest available evidence the findings come from the skills as shipped, and highlights the
specific points where human judgment intervened.

## 4. Outside data used

Beyond the **provided corpus** (LDA filings + congressional press releases), the findings use
these **public** outside sources:

- **USAspending.gov** (`api.usaspending.gov` federal award API) — the money-trail dollar
  figures in Finding 02. Reproducible via the `federal-award-tracer` skill / `skills/federal-award-tracer/scripts/04_award_tracer.py`.
- **Public biographical / role-confirmation sources** — used only to independently confirm each
  subject's prior agency role and departure date: agency sites (energy.gov, cftc.gov, usda.gov,
  fcc.gov, doi.gov/bia.gov), LegiStorm, the subjects' own firm websites, the Center for
  Presidential Transition, and news outlets (Bloomberg Law, Law360, FIA). Every URL is listed in
  `notes/08` and `notes/09`.

No paid datasets, no private data, no scraping of non-public sources.

## 6. Possible legal violations flagged to the panel

**No finding alleges a legal violation, and we want to be explicit about that.** Our reportability
analysis (`notes/09_reportability_gates_207_and_comment.md`) examined every top case against the
federal post-employment statute, **18 U.S.C. §207**, and reached the following:

- **The timed "cooling-off" bans (§207(c) one year; §207(d) two years) are cleared in every
  examined case.** Each official's observed lobbying of their former agency post-dates their
  departure by years — from ~3 years (Barbic, the tightest case) to ~18 years (Newsome). On the
  evidence, **the lobbying we document is legal on its face.** That legality is precisely what
  makes the *structure and scale* the story, not a prosecution.
- **The permanent particular-matter bar (§207(a)(1)) is flagged as an open question, not a
  violation we assert.** It bars a former official from lobbying on the *specific matters they
  personally handled in government*, and cannot be established from lobbying-concentration data
  alone. We raise it as a question to put to each subject in the request-for-comment stage — not
  as an allegation.

So: **nothing here should be read as a claim of criminal conduct.** If the panel's fact-checking
were to establish a specific §207(a)(1) matter overlap for any subject, that would warrant separate,
much stronger substantiation before publication; we have not made — and the data does not support —
such a claim.

## Reproducing

**This package is self-contained.** Each skill ships with its own `scripts/`; alongside them are
`tests/` (the pytest suite), `tools/` (the AP-style linter — a supporting tool, not a skill),
`findings-report.md`/`.pdf`, `notes/` (finding-support sourcing), `traces/`, and pinned
dependencies (`pyproject.toml` + `uv.lock`). From the package root:

```bash
uv sync --extra dev          # install pinned dependencies (Python ≥ 3.11, uv)
uv run skills/doctor/scripts/doctor.py     # validate setup (exits 0 with guidance if corpus/DB absent)
uv run python -m pytest      # 250 pass, 11 skip (DB-dependent) with no DB present
```

*Windows note:* extract to a short path (e.g. `C:\fairguard`) or enable long-path
support. Some dependencies (e.g. `networkx`) have deeply nested modules, and installing
them under a very long extraction path can silently hit the legacy 260-character
`MAX_PATH` limit during `uv sync`.

To run the finding-producing skills you need `output/investigation.duckdb`, which is built from
the 8.6 GB corpus (**not** shipped in this zip — it exceeds submission size limits). Two ways to
obtain it:

- **Fast (~10 min):** download the pre-built `output/` from Google Drive, unzip at the package
  root: https://drive.google.com/drive/folders/1O_qsxmFitgRfyjPXsgyDSjrbX3L-1Vlf?usp=sharing
- **Full build (~2.5 hr):** place the raw challenge corpus in `data/` and run the
  `lda-corpus-indexer` skill (`uv run skills/lda-corpus-indexer/scripts/01_build_index.py`).

*(The source is also mirrored on GitHub — https://github.com/MokshitSurana/northwestern-challenge —
but nothing in this submission requires it; the zip stands alone.)*

Then, e.g.: `uv run skills/revolving-door-detector/scripts/03_agency_concentration.py` (scan) →
`uv run skills/federal-award-tracer/scripts/04_award_tracer.py --case skills/federal-award-tracer/cases/steinberg_doe.json` (trace).

> **Note on the GitHub mirror:** the mirror keeps a single shared top-level `scripts/` library
> that all skills call (its dev/monorepo layout). This submission package instead reorganizes
> each skill to be self-contained (`skills/<name>/scripts/…`) to match the recommended structure,
> and updates the invocation paths accordingly. Same code, same results — only the paths differ.
